# Clinic Info, Services, and FAQ Answering Flow

This document explains how the current agent answers user questions about:

- service details
- pricing
- insurance
- hours
- location and parking
- staff
- payment methods
- other clinic FAQs

It covers:

- how FAQ data is stored in Supabase
- how the platform writes and reads that data
- how the voice runtime fetches and uses it
- how rule-based and deterministic retrieval work
- where the optional LLM phrasing layer is inserted
- what is and is not generated by the LLM

## 1. High-Level Architecture

The current FAQ and service-answering system is a hybrid of:

1. Supabase-backed knowledge storage
2. deterministic question detection and answer composition
3. an optional final LLM phrasing pass for more natural spoken output

The important design point is:

- fact retrieval is deterministic
- fact selection is deterministic
- article scoring is deterministic
- service scoping is deterministic
- the optional LLM is only used to rewrite an already-verified answer into more natural spoken English

So this is not a classic runtime RAG chatbot that sends the whole query to an LLM and asks it to search.

## 2. Supabase Data Model

Clinic FAQ content is stored in the `knowledge_articles` table.

Current schema, based on the project architecture docs:

- `id`: UUID primary key
- `organization_id`: UUID foreign key
- `clinic_id`: UUID foreign key
- `title`: article title
- `category`: category such as `FAQ`, `Services`, `Insurance`, `Payment`, `Staff`, `Policy`
- `body`: full content
- `status`: article lifecycle such as `active`, `draft`, `processing`
- `search_vector`: PostgreSQL `tsvector` for full-text search on the platform side

The key mappings are:

- `organization_id` scopes ownership and access control
- `clinic_id` scopes the article to one clinic
- the voice runtime loads articles by `clinic_id`

This means FAQ content is clinic-specific, not global.

## 3. How Articles Get Into Supabase

The platform writes FAQ records into `knowledge_articles`.

There are two main ways this happens:

1. Platform onboarding
2. Knowledge base management endpoints

### 3.1 Onboarding Flow

The platform README and onboarding routes show that onboarding creates linked records across:

- `organizations`
- `clinics`
- `agents`
- `agent_settings`
- `knowledge_articles`
- `phone_numbers`

So a newly onboarded clinic can be provisioned with starter FAQs and service information up front.

### 3.2 Knowledge Management API

The platform backend exposes routes in `agent_platform/backend/src/routes/knowledge.js`.

These routes:

- list articles for a clinic
- create an article
- update an article
- delete an article
- search articles
- import scraped content from a URL

Every query is scoped by both:

- `clinic_id`
- `organization_id`

So the platform layer enforces tenant isolation before the voice runtime ever sees the content.

## 4. Platform Search vs Runtime Search

There are two different search concepts in the system:

### Platform Search

The platform backend uses Supabase/Postgres full-text search on `search_vector`.

This is mainly for:

- admin UI search
- article editing workflows
- finding relevant articles in the platform dashboard

### Runtime Voice Answering

The voice runtime does not use Supabase text search during the live call.

Instead, `agent.py` fetches the clinic's knowledge articles at call startup and keeps them in memory.

From that point onward, live answering is local and deterministic.

This avoids:

- an extra network trip per FAQ question
- extra lookup latency during the call
- variable retrieval timing in the middle of a voice turn

## 5. Runtime Load Path in `agent.py`

At call/session startup, the runtime fetches knowledge articles with:

- `select("title, body, category")`
- `eq("clinic_id", clinic_id)`
- `limit(20)`

That fetch is done once per clinic context load.

The returned rows are normalized into:

```python
{
  "title": "...",
  "body": "...",
  "category": "..."
}
```

The runtime then uses the articles in two different ways:

1. index-only prompt context for the main conversational system prompt
2. full deterministic answer composition in `AssistantTools`

## 6. Two Separate Uses of FAQ Data

This part is important because there are two FAQ consumers in the current system.

### 6.1 Prompt Context: `clinic_faq`

`agent.py` builds a compact prompt summary using `_format_clinic_faq(...)`.

This summary only includes:

- article category
- article title

It does not include full article bodies.

Example shape:

```text
- [Pricing] Teeth whitening pricing
- [Insurance] PPO plans accepted
- [Parking] Parking instructions
```

This compact FAQ index is injected into the main system prompt so the main LLM has general awareness of what information exists.

Important:

- this prompt context is intentionally lightweight
- it is not the primary answering mechanism for service/pricing/FAQ questions
- the current direct answering path uses deterministic tool logic instead

### 6.2 Deterministic Answering: `knowledge_articles`

The full normalized article bodies are passed into `AssistantTools`.

Those full articles power:

- `search_clinic_info(...)`
- `answer_clinic_question(...)`
- TTS sanitization of clinic answers

This is the main live FAQ answering path.

## 7. How a User Question Gets Routed to Clinic Info Answering

There are two main places where FAQ answering can be triggered.

### 7.1 Main Conversation Fast Path

The turn-taking pipeline can classify a user turn into the deterministic route:

- `clinic_info.answer`

Examples:

- "What is the pricing of teeth whitening?"
- "Do you take Delta Dental?"
- "What payment methods do you accept?"

When that route is selected, `agent.py` calls:

```python
assistant_tools.answer_clinic_question(question_text, include_follow_up=...)
```

This bypasses the general freeform LLM response path and gives a direct answer quickly.

### 7.2 Post-Booking Interruption Handling

Even after booking, the runtime can still intercept clinic-info questions deterministically.

For example:

- after the appointment is booked
- while asking delivery preference
- during the "anything else I can help with" phase

In those states, the agent checks:

```python
assistant_tools.can_answer_clinic_question(text)
```

If true, it answers via the same deterministic clinic-info path rather than using a generic LLM reply.

## 8. Rule-Based Question Detection

The first layer is rule-based detection of whether a question is a clinic-info question at all.

This logic lives in:

- `_looks_like_clinic_info_question(...)`
- `_question_knowledge_categories(...)`
- regex-based hint matchers

### 8.1 Keyword Families

The runtime uses explicit regex hint groups, such as:

- pricing
- insurance
- hours
- location
- parking
- payment
- staff
- policy
- emergency
- logistics
- service information

There are also spa-specific patterns for:

- memberships
- pre-care
- post-care
- contraindications
- results
- consultation
- gratuity
- cancellation
- arrival

### 8.2 Why This Matters

This rule-based layer decides:

- whether the question should be treated as clinic info
- which categories it belongs to
- which scoring boosts or penalties should apply later

So the system does not ask the LLM to decide whether something is a pricing question.
That mapping is done locally and deterministically.

## 9. Service Context Injection for Follow-Up Questions

The system also handles short follow-up questions like:

- "What type is better?"
- "How much is it?"
- "How long does it take?"

If the caller already has a known service in state, the runtime may rewrite the question internally to add service context.

For example:

- user asks: `What type is better?`
- stored service: `Teeth whitening`
- internal contextualized form becomes something like: `What type is better about Teeth whitening`

This helps short follow-up questions stay scoped to the already known service.

This is still deterministic and local.

## 10. Deterministic Retrieval and Ranking

Once the runtime decides the question is clinic info, it composes the answer using local article ranking.

This logic lives mostly in:

- `_question_topic_terms(...)`
- `_question_specific_terms(...)`
- `_question_knowledge_categories(...)`
- `_knowledge_match_score(...)`
- `_rank_knowledge_articles(...)`
- `_select_knowledge_articles_for_answer(...)`
- `_compose_knowledge_answer(...)`

### 10.1 Terms Extracted from the Question

The runtime builds a set of question terms using:

- cleaned word tokens
- detected service terms
- category-specific hint terms

For example, a pricing question can add topic terms like:

- `price`
- `pricing`
- `cost`
- `fee`
- `rate`

A payment question can add:

- `payment`
- `cash`
- `visa`
- `card`
- `carecredit`

### 10.2 Category-Aware Scoring

Each article gets a deterministic score based on:

- title term matches
- body term matches
- category term matches
- question category boosts
- pricing value boosts
- service-specific boosts
- penalties when pricing/services are requested but no specific service is matched

This makes article ranking explainable and stable.

### 10.3 Service-Specific Filtering

When the question clearly refers to a service like:

- teeth whitening
- root canal
- night guard

the selection logic filters out articles that do not actually contain that service.

This prevents answers such as:

- whitening question returning root canal pricing
- whitening question returning night guard details

## 11. Deterministic Answer Composition

After ranking, the runtime composes a spoken answer from the selected articles.

Key behaviors:

### 11.1 Top Article Selection

The system selects the best articles, usually a small number, not the entire knowledge bank.

### 11.2 Sentence Extraction

If an article body has multiple sentences, the runtime tries to extract the most relevant sentences.

For example:

- service-matching sentences get priority
- pricing sentences get boosted for pricing questions
- follow-on pricing sentences may be included when they stay on the same service

### 11.3 Voice-Length Compression

Answers are compressed for voice delivery:

- capped to short spoken length
- normalized into sentence form
- trimmed to avoid overly long brochure-style responses

### 11.4 Final Service Safety Net

There is a final pruning pass that removes sentences mentioning other services if the caller asked about one specific service.

This is a very important safety layer.

Even if article ranking or sentence extraction lets unrelated text through, the final output is pruned again before being used.

## 12. Clarification and Fallback Behavior

If the caller asks for pricing or services but the requested treatment is unclear, the deterministic flow does not guess.

Instead it returns a clarification prompt such as:

```text
I want to make sure I give you the right pricing. Which treatment would you like details for?
```

If the question category is known but no exact answer is found, deterministic category-specific fallbacks are used, for example:

- pricing fallback
- insurance fallback
- hours fallback
- location fallback
- generic fallback

These are still rule-based, not LLM-generated.

## 13. The `search_clinic_info(...)` Tool

`search_clinic_info(question)` is the direct clinic-info lookup tool.

Current behavior:

1. deterministic answer composition runs first
2. if a deterministic answer exists, it is returned
3. if no answer exists, a fallback message is returned
4. before returning, the answer may go through the optional speech-humanizer layer

This tool is meant for:

- pricing
- hours
- insurance
- location
- parking
- service details
- other structured clinic FAQs

## 14. The `answer_clinic_question(...)` Helper

`answer_clinic_question(question, include_follow_up=False)` is the higher-level helper used by the runtime.

It does:

1. check whether the utterance is a clinic-info question
2. compose the deterministic answer
3. humanize it for spoken output
4. optionally append:

```text
Is there anything else I can help you with today?
```

That is why the same core FAQ logic can work:

- mid-call
- post-booking
- during delivery-preference handling

## 15. TTS Sanitization Layer

There is an additional protection layer before speech playback:

- `agent.py` calls `_sanitize_spoken_output_for_tts(...)`
- that may call `prune_clinic_response_for_tts(...)`

This is specifically there to stop spoken answers from drifting into unrelated service content.

So even if a longer answer somehow reaches the speech layer, the TTS sanitizer can still prune it back to the requested service scope.

## 16. Where the LLM Is Used

The LLM is not used to retrieve facts from Supabase.

It is also not used to rank articles.

It is also not used to choose which article is correct.

Those parts are deterministic.

The optional LLM is only inserted after a deterministic answer already exists.

### 16.1 The LLM Humanizer Layer

The speech-humanizer lives in:

- `humanize_clinic_answer_for_voice(...)`
- `_humanize_clinic_answer_with_prompt(...)`

Its purpose is:

- make spoken answers sound warmer
- turn label-style answers into natural speech
- keep facts unchanged

### 16.2 Rule-Based First, LLM Second

The system first tries a rule-based humanizer.

This handles compact patterns like:

```text
Teeth whitening: $280 (30 minutes).
```

and rewrites them locally into:

```text
Teeth whitening is $280, and it usually takes about 30 minutes.
```

This path does not use an LLM.

Only if the answer is not covered by the rule-based formatter does the optional prompt-based humanizer run.

## 17. How the LLM Humanizer Prompt Works

If enabled, the system builds a tiny standalone prompt with structured facts extracted from the deterministic answer:

- caller question
- topic
- service name
- pricing
- time period
- verified answer

The system prompt instructs the model to:

- keep every fact exactly the same
- not add or remove details
- rewrite into one short spoken sentence, or two if needed
- avoid bullets, markdown, greetings, and sign-offs

This is a small isolated LLM call, not the main conversation model path.

### 17.1 Current Runtime Settings

The current humanizer uses:

- model: `gpt-4o-mini` by default
- temperature: `0.2`
- max completion tokens: `80`
- timeout: `1.35s`

It also keeps an in-memory cache of formatted outputs so repeated FAQ answers can be cheaper and faster.

## 18. Fact Preservation Guard

The optional LLM humanizer is not trusted blindly.

After it returns, the runtime validates that important facts are still present.

Currently this guard checks for preservation of:

- price values
- timing/duration values

If the rewritten answer drops or changes those extracted facts, the runtime discards the LLM rewrite and falls back to the deterministic answer.

So the LLM is allowed to improve phrasing, but not to redefine the facts.

## 19. When the LLM Humanizer Is Skipped

The prompt-based humanizer is skipped when:

- the rule-based formatter already handled the answer
- the answer is a deterministic fallback like "I don't have that exact detail..."
- tests are running
- `CLINIC_INFO_HUMANIZER_ENABLED=0`
- the prompt-based formatter errors
- the formatter times out

In all of those cases, the runtime simply returns the deterministic answer.

## 20. End-to-End Example: Pricing Question

Example user question:

```text
Can you tell me the pricing of teeth whitening?
```

Current flow:

1. turn classifier routes the utterance to `clinic_info.answer`
2. `assistant_tools.answer_clinic_question(...)` is called
3. the question is recognized as a pricing + services question
4. service extraction detects `Teeth whitening`
5. knowledge articles already loaded in memory are ranked
6. relevant whitening article(s) are selected
7. unrelated services are filtered out
8. deterministic answer text is composed
9. if the answer is compact and pattern-based, the rule-based humanizer rewrites it locally
10. otherwise, the optional LLM humanizer may rewrite only the phrasing
11. TTS sanitization can still prune if needed before speech
12. the agent speaks the final answer

## 21. End-to-End Example: General FAQ

Example user question:

```text
What payment methods do you accept?
```

Flow:

1. rule-based category detection maps this to `payment`
2. matching payment article(s) are ranked
3. insurance articles are not preferred for this question
4. deterministic answer is composed from payment content
5. optional LLM humanizer may rewrite it into a more natural spoken sentence
6. the answer is spoken directly

## 22. What Is Deterministic vs What Is LLM-Based

### Deterministic

- Supabase fetch of clinic knowledge articles
- question category detection
- service detection
- article scoring
- article ranking
- service filtering
- sentence selection
- clarification prompts
- fallback answers
- TTS pruning of unrelated service content

### Optional LLM-Based

- final speech phrasing only

The LLM does not decide which article is true.

## 23. Current Tradeoffs

The current design gives a good balance for voice:

### Strengths

- low latency after call startup
- no per-question database lookup
- deterministic service scoping
- easier debugging than full RAG
- safer fact handling than freeform LLM answers
- optional natural phrasing when useful

### Limitations

- runtime currently loads up to 20 articles for the clinic
- prompt context uses title/category only, not full bodies
- deterministic ranking depends on current regexes and service extraction quality
- the optional humanizer currently validates prices and timing, but not every possible fact type

## 24. Practical Summary

If you want to think about the current system in one sentence:

The agent answers service and FAQ questions by loading clinic-specific articles from Supabase once per call, selecting and composing answers with deterministic rule-based logic, and then optionally running a very small guarded LLM pass only to make the final spoken sentence sound more human.
